# 智能体意图识别子模块 - 历史会话结合策略

核心结论：意图识别子智能体必须结合「历史会话+当前输入」一起分析，才能精准捕捉用户的真实意图。只看单轮输入会丢失上下文，导致意图识别片面、甚至完全错误。

# 一、为什么必须传递历史会话？

用户的真实意图往往是「跨轮次」的，单轮输入可能只是“补充指令”或“追问”，脱离历史就会理解偏差。

## 1.1 典型场景对比（直观体现历史会话的必要性）

|场景|仅传当前输入的问题|结合历史的正确意图|
|---|---|---|
|用户轮1：“查上周销售数据”用户轮2：“算环比增长率”|识别为“无上下文的增长率计算”，调用计算器工具|识别为“基于上周销售数据计算环比增长率”，调用数据统计工具|
|用户轮1：“生成销售报告”用户轮2：“加上图表”|识别为“生成图表”，调用绘图工具|识别为“补充销售报告的图表需求”，调用报告工具+绘图工具|
|用户轮1：“帮我分析数据”用户轮2：“不对，是运营数据”|识别为“分析运营数据”（无历史则无法修正）|识别为“修正之前的需求：分析运营数据而非销售数据”|
## 1.2 核心价值

- 修正偏差：用户后续轮次可能纠正之前的模糊表述；

- 补充上下文：用户不会在每轮都重复核心需求（如“销售数据”）；

- 理解递进需求：用户的需求是逐步细化的（先要报告，再要图表，最后要导出PDF）；

- 避免重复操作：识别出用户是“追问结果”而非“重新执行”，减少智能体无效调用。

# 二、该传递哪些历史会话？（不是越多越好）

传递全部历史会增加Token成本、降低识别效率，需筛选「有效历史」，核心原则：**兼顾准确性与成本**。

## 2.1 筛选维度

1. 时间范围：近N轮（如近5-10轮），超过一定轮次的历史对当前意图无意义；

2. 内容相关性：过滤掉无关闲聊（如“你好”“谢谢”），只保留和核心需求相关的内容；

3. 角色过滤：保留「用户说的话」+「智能体的关键回复」（如“已为你调取销售数据”），过滤智能体的无关提示。

## 2.2 推荐传递格式（结构化）

不要传原始文本，按「角色+内容+时间戳」结构化传递，方便意图识别子智能体解析（JSON格式示例）：

```json

[
  {"role": "user", "content": "查上周销售数据", "timestamp": 1710000000},
  {"role": "agent", "content": "已为你调取2026年第10周销售数据：总销售额100万", "timestamp": 1710000060},
  {"role": "user", "content": "算环比增长率", "timestamp": 1710000120}
]
    
```

## 2.3 关键阈值建议

- 轮次上限：5-10轮（根据模型Token限额调整，如GPT-3.5建议≤8轮）；

- 内容长度：单轮内容≤200字（过长则做摘要，比如“用户上传了销售数据表格，要求分析趋势”）；

- 过滤规则：剔除纯寒暄、无实质信息的内容（可通过关键词/长度判断，如仅“好的”“谢谢”则过滤）。

# 三、怎么把历史+当前输入传给意图识别子智能体？

核心是「构造清晰的Prompt」，让子智能体明确“要结合历史分析当前意图”，以下是可直接落地的方案。

## 3.1 Prompt模板（核心逻辑，通用版）

```text

你是智能体的意图识别子模块，需要结合用户的历史会话和当前输入，分析出用户的真实、可执行的核心意图。

## 输入信息
1. 历史会话（按时间从早到晚）：
{history_messages}

2. 当前用户输入：
{current_input}

## 输出要求
1. 输出格式：JSON
2. 输出字段：
   - core_intent：真实核心意图（简洁、明确、可执行，如“基于上周销售数据计算环比增长率”）
   - intent_type：意图类型（如“数据计算”“数据查询”“报告生成”“需求修正”）
   - context_keywords：上下文关键词（如“上周销售数据、环比增长率”）
   - is_correction：是否为需求修正（true/false）
   - is_continuation：是否为上一轮需求的延续（true/false）

## 注意事项
1. 必须结合历史会话分析，不能仅看当前输入；
2. 若用户是修正之前的需求，需在core_intent中明确修正后的内容；
3. 若当前输入是无意义闲聊，core_intent填“无实质需求”；
4. 核心意图必须是“可执行的”，而非模糊描述（如避免“帮我处理数据”，要写“分析上周销售数据的环比增长率”）。
    
```

{history_text}

    """

# 四、关键落地细节（避坑点）

## 4.1 处理“意图修正”场景

用户可能说“我刚才说错了，不是销售数据，是运营数据”，此时意图识别子智能体需要：

- 标记`is_correction: true`；

- `core_intent`中明确修正后的意图（而非保留原意图）；

- 后续的意图分解/工具选择模块，需基于修正后的意图执行（而非重复原错误逻辑）。

## 4.2 处理“无实质需求”场景

用户输入“你好”“谢谢”“随便看看”时，直接标记`core_intent: 无实质需求`，后续模块无需执行，避免智能体无效调用工具。

## 4.3 历史消息的“轻量化”

如果历史消息过长（如用户上传了大段文本），先对历史消息做「摘要」（比如调用LLM生成100字以内的摘要），再传入意图识别子智能体，减少Token消耗和识别耗时。

## 4.4 意图识别的“缓存与复用”

如果用户连续多轮的意图未变化（如“生成报告”→“报告加上图表”→“报告导出为PDF”），可缓存核心意图，仅更新补充信息，避免重复调用意图识别子智能体。

# 五、总结

1. 必须传历史会话：用户真实意图是跨轮次的，仅看当前输入会导致识别偏差，核心解决“上下文缺失”问题；

2. 传“有效历史”：限制轮次、过滤无关内容、结构化传递，平衡识别准确性和成本；

3. Prompt明确要求：让意图识别子智能体清晰知道“要结合历史分析”，并输出结构化的意图结果，便于后续模块对接；

4. 处理特殊场景：重点关注“需求修正”“无实质需求”，避免智能体做无效操作。

按此策略设计后，意图识别子智能体可输出精准、可执行的真实意图，为后续的意图分解、工具选择打下坚实基础。
